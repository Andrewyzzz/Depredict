# 集体AI审议：基于多专家咨询的人类观点精炼框架

**版本 1.0 | 2026年3月**

---

## 摘要

本文提出一种多智能体AI咨询框架，通过用户与领域专属AI专家克隆体的结构化交互，帮助用户精炼和修正其观点。与倾向于迎合用户的单一大语言模型（LLM）聊天机器人不同，我们的系统通过组建由5位具有多元分析视角的AI专家组成的咨询团，创造**结构化认知冲突**。每位专家均通过领域专属知识档案（Level 2 AI克隆技术）进行增强。系统采用4阶段咨询流程——独立评估、交互对话、专家讨论、最终综合——识别用户思维中的盲点，挑战认知偏差，并产出可操作的建议。我们以加密货币投资分析和体育预测两个领域的实现来展示该框架。本方法融合了机制设计、检索增强生成（RAG）和多智能体系统的技术，创造出比单一AI助手更平衡、更严谨、更具信息量的咨询体验。

## 1. 引言

### 1.1 问题背景

大语言模型（LLM）已成为寻求建议和分析的默认工具。然而，单一LLM交互存在几个已被充分记录的局限性：

1. **迎合性（Sycophancy）**：LLM倾向于同意用户陈述的立场，强化现有偏见而非挑战它们（Perez et al., 2023; Sharma et al., 2023）。

2. **视角同质化**：单个模型只能提供一种分析镜头。复杂决策——投资策略、职业选择、政策判断——需要多元视角，这不是单个系统提示词能充分覆盖的。

3. **缺乏结构化分歧**：在人类专家咨询中（如医疗会诊、投资委员会决策），专家间的分歧是特性而非缺陷。分歧能暴露假设、识别风险、产出更校准的判断。单一LLM交互缺乏这种机制。

4. **浅层领域专业性**：通用LLM缺乏专业分析框架、典型推理模式和领域特定启发式方法——这些是真正领域专家的标志。

### 1.2 我们的方法

我们提出**集体AI审议**框架，通过三个核心创新解决上述局限：

- **Level 2 AI克隆**：每个专家智能体通过领域专属知识档案进行增强，包含分析框架、少样本推理示例和特征性表达模式。这将通用LLM转化为具有一致分析行为的专业专家克隆体。

- **多阶段咨询流程**：一个结构化的4阶段流程，分离独立评估（减少群体思维）、交互对话（促进用户参与）、同行讨论（强制交叉审查）和最终综合（产出精炼结论）。

- **设计化多样性**：专家面板以有意设计的多样化分析立场（支持、质疑、审慎、中立）构建，确保多元视角得到表达，防止回音室效应。

### 1.3 贡献

1. 提出一种新颖的AI专家克隆方法论（Level 2），使用结构化知识档案和少样本示例，从通用LLM创建领域专业化智能体。
2. 设计了一个包含用户参与的多阶段咨询流程。
3. 实现了两个完整的领域Demo（加密货币和体育），验证框架的适用性。
4. 提供基于Streamlit的开源参考实现。

## 2. 相关工作

### 2.1 多智能体LLM系统

近期工作探索了使用多个LLM智能体改善推理能力。**LLM Debate**（Du et al., 2023）表明多智能体辩论可改善数学和事实推理。**Society of Mind**（Zhuge et al., 2023）展示了智能体角色专业化的优势。**ChatEval**（Chan et al., 2023）将多智能体辩论用于文本评估。我们的工作将这些方法从事实/推理任务扩展到**观点精炼**领域，其目标不是单一正确答案，而是更校准、更深思熟虑的判断。

### 2.2 AI辅助决策

**超级预测**研究（Tetlock & Gardner, 2015）证明了结构化分析框架和多元视角能提高预测准确性。**德尔菲法**使用带有可控反馈的迭代专家咨询。我们的系统以计算方式实现了这些原则，用AI智能体替代人类专家，同时保留结构化分歧机制。

### 2.3 预测市场与机制设计

我们的框架建立在预测市场的机制设计原则之上。**LMSR**（Hanson, 2003）提供了一种有原则的信念聚合方法。**贝叶斯真相血清（BTS）**（Prelec, 2004）在无需真值的情况下激励诚实报告。**声誉系统**跟踪专家随时间变化的可靠性。我们将这些机制适配到观点咨询的场景中。

### 2.4 检索增强生成

**RAG**（Lewis et al., 2020）将LLM回答锚定在检索到的证据上。**RAFT**（Zhang et al., 2024）训练模型区分相关和不相关文档。我们的系统使用RAG为专家提供时效性信息，同时利用RAFT风格的文档评估来过滤噪音。

```
└──────────────────────┬───────────────────────────┘
                       │
┌──────────────────────▼───────────────────────────┐
│               知识与克隆层                        │
│    知识档案: 分析框架 + 少样本示例 + 表达风格      │

└──────────────────────┬───────────────────────────┘
                       │
┌──────────────────────▼───────────────────────────┐
│                信息检索层                          │
│    RAG: YouTube字幕 + Tavily新闻搜索              │
│    RAFT风格文档混合与评估                         │
│    信息分区实现不对称性                            │
└──────────────────────────────────────────────────┘
```

### 3.2 设计原则

**原则1：结构化分歧优于共识。** 系统的设计目标是暴露分歧而非压制分歧。专家立场预先分配，以确保多元观点。

**原则2：用户是参与者而非旁观者。** 与自动化预测系统不同，用户在第2阶段积极与专家互动，既挑战专家也被专家挑战。

**原则3：基于知识的专业性。** 每位专家的分析行为锚定在具体的知识档案上，而非仅仅是人设描述。

**原则4：并行执行提升效率。** 所有阶段使用并行执行（`ThreadPoolExecutor`），最小化延迟。

## 4. Level 2 AI克隆技术

### 4.1 克隆层级体系

我们定义了三个级别的AI专家克隆：

| 级别 | 方法 | 人格深度 | 成本 |
|------|------|---------|------|
| **Level 1** | 系统提示词人设（名字 + 描述） | 浅层：同一个LLM戴不同帽子 | 低 |
| **Level 2** | 知识档案注入（框架 + 少样本 + 风格） | 中层：一致的分析行为模式 | 中 |
| **Level 3** | 微调模型（基于专家语料的LoRA适配器） | 深层：内化的专业知识 | 高 |

Level 1 是大多数多智能体系统采用的方法——它提供表面的角色区分，但智能体很快会收敛到相似的推理模式。Level 3 需要大量训练数据和算力。**Level 2 是实用的最佳平衡点**：提供有意义的行为差异化，同时不需要微调的成本。

### 4.2 知识档案结构

每位专家的知识档案是一个结构化的Markdown文档，包含：

```
# 专家名称 — 知识档案

## 原型
该克隆体所参照的真实专家/学派。

## 核心分析框架
### 关键分析维度
专家主要分析工具和指标的编号列表。

### 核心原则/信念
专家的基本假设和偏好。

## 少样本示例

### 示例1：[场景类型]
**问题/用户观点**: [输入]
**分析**: [展示框架使用的完整专家风格回应]

### 示例2：[场景类型]
...

### 示例3：[场景类型]
...

## 典型表达与风格
保持一致性的特征性表达方式。
```

### 4.3 提示词架构

`build_clone_prompt()` 函数组装一个4层系统提示词：

```
┌─────────────────────────────────┐
│ 第1层：身份定义                  │  专家名称、角色、专业领域
├─────────────────────────────────┤
│ 第2层：知识档案                  │  约占总提示词的80%
│  - 分析框架                     │  框架、示例、风格特征
│  - 3个少样本示例                │  从Markdown文件加载
│  - 典型表达                     │
├─────────────────────────────────┤
│ 第3层：行为规则                  │  保持角色、使用框架、
│                                 │  诚实建议、给出具体操作
├─────────────────────────────────┤
│ 第4层：RAG上下文                │  当前话题的检索文档
└─────────────────────────────────┘
```

知识档案约占系统提示词的80%，使专家的"分析DNA"成为LLM上下文窗口中的主导信号。

## 5. 多阶段咨询流程

### 5.1 流程总览

```
用户输入：话题 + 观点 + 领域
         │
         ▼
    ┌──────────┐     RAG检索：YouTube字幕
    │ 上下文检索 │──── + 新闻文章，RAFT风格混合
    │          │     + 信息分区
    └────┬─────┘
         │
         ▼
    ┌──────────┐     5位专家独立评估
    │ 第1阶段   │──── 并行执行（ThreadPoolExecutor）
    │ 独立评估  │     输出：评价、认同/质疑点、
    └────┬─────┘     关键洞察、置信度评分
         │
         ▼
    ┌──────────┐     用户阅读评估结果
    │ 第2阶段   │──── 用户与专家一对一对话
    │ 交互对话  │     维护对话历史
    └────┬─────┘     用户满意后继续
         │
         ▼
    ┌──────────┐     每位专家看到其他所有专家的评估
    │ 第3阶段   │──── 专家回应、更新评估、识别盲点
    │ 专家讨论  │     并行执行
    └────┬─────┘
         │
         ▼
    ┌──────────┐     每位专家给出最终评估
    │ 第4阶段   │──── 具体建议 + 风险警示
    │ 最终评估  │     置信度追踪（第1阶段 → 第4阶段）
    └────┬─────┘
         │
         ▼
    ┌──────────┐     聚合洞察、盲点、
    │ 报告综合  │──── 建议、风险警示
    │          │     置信度变化可视化
    └──────────┘
```

### 5.2 第1阶段：独立评估

每位专家在看不到其他专家观点的情况下独立评估用户的观点。这能防止锚定效应和群体思维。

**输入**：话题、用户观点、RAG上下文（通过信息分区实现专家专属）

**输出结构**：
- `ASSESSMENT`：整体评价（支持/部分支持/反对）
- `AGREE_POINTS`：专家认同的部分及理由
- `CHALLENGE_POINTS`：专家质疑的部分及证据
- `KEY_INSIGHT`：用户最应该知道的一个关键信息
- `CONFIDENCE`：0-100%的置信度评分

**设计理由**：通过要求结构化的认同和质疑，我们防止了迎合性失败模式。即使专家大体同意，也必须找出可质疑的点。

### 5.3 第2阶段：交互对话

用户阅读所有第1阶段评估后，可以与任何专家进行自由对话。这是**用户参与闭环**组件，区分我们的系统与全自动化多智能体辩论。

**核心特性**：
- 每位专家维护独立的对话历史
- 专家回应具有上下文感知（话题 + 原始观点 + 对话历史）
- 无强制结构——用户驱动对话深度

**设计理由**：用户往往有追问需求或想挑战专家的具体观点。这个阶段允许用户测试专家推理的鲁棒性。

### 5.4 第3阶段：专家讨论

专家审阅彼此的第1阶段评估，进行结构化的同行讨论。

**输入**：每位专家看到所有其他专家的评估（不包括自己的——他们已经有了）

**输出结构**：
- `RESPONSES`：针对每位专家的定向回应（"回应 [专家名]：[内容]"）
- `UPDATED_ASSESSMENT`：考虑同行观点后的修正评价
- `BLIND_SPOTS`：尚未被充分讨论的议题
- `CONFIDENCE`：更新后的置信度

**设计理由**：专家间讨论强制交叉审查。过度自信的专家必须回应同行的挑战。遗漏因素的专家必须承认不足。这模拟了真实专家委员会的动态。

### 5.5 第4阶段：最终评估

在完整讨论之后，每位专家给出整合第1-3阶段所有信息的最终评估。

**输出结构**：
- `FINAL_ASSESSMENT`：2-3句话的结论性评价
- `RECOMMENDATION`：具体、可操作的建议
- `RISK_WARNING`：1-2个用户应监控的关键风险
- `CONFIDENCE`：最终置信度评分

### 5.6 报告综合

系统将所有第4阶段输出聚合为结构化咨询报告：

1. **专家面板概要**：每位专家的最终评估
2. **关键洞察**：每位专家从其领域提供的独特见解
3. **盲点识别**：讨论过程中识别出的分析盲区
4. **行动建议**：来自所有专家的可操作建议
5. **风险警示**：共识性和分歧性的风险评估
6. **置信度变化**：专家置信度从第1阶段到第4阶段的变化趋势

## 6. 信息检索与知识锚定

### 6.1 RAG管道

系统检索时效性信息以锚定专家分析：

1. **YouTube字幕检索**：搜索与话题相关的视频，通过 `YouTubeTranscriptApi` 提取完整字幕。更长的视频（意味着更深入的分析）被优先排列。

2. **新闻搜索**：Tavily API 提供近期新闻文章作为补充上下文。

3. **RAFT风格文档混合**：文档被分为"oracle"（高质量、相关）和"distractor"（可能不相关）两类，然后打乱混合。专家必须在分析过程中评估文档相关性。

### 6.2 信息分区

为在专家之间创造真实的信息不对称：

```
全部文档
    ├── 40% → 共享池（所有专家都能看到）
    └── 60% → 私有池
                ├── 专家1：2篇独占文档
                ├── 专家2：2篇独占文档
                ├── 专家3：2篇独占文档
                ├── 专家4：2篇独占文档
                └── 专家5：2篇独占文档
```

这确保专家拥有不同的信息基础，使其独立评估是真正多元的，而非表面多元。

## 7. 技术实现

### 7.1 系统要求

- **Python 3.11+**
- **LLM后端**：DeepSeek V3（主要），可选 OpenAI GPT-4o、Anthropic Claude、Google Gemini
- **API**：Tavily（新闻搜索）、YouTube Data API（字幕检索）
- **前端**：Streamlit + Plotly 可视化
- **执行**：ThreadPoolExecutor 并行执行

### 7.2 核心组件

| 组件 | 文件 | 职责 |
|------|------|------|
| `ConsultAgent` | `consult_agent.py` | 单个专家智能体，含4阶段方法 |
| `ConsultationPipeline` | `consultation.py` | 流程编排与报告综合 |
| `build_clone_prompt` | `knowledge_loader.py` | Level 2 知识档案注入 |
| 专家面板 | `expert_panels.py` | 领域专属专家配置 |
| `NewsRetriever` | `retriever.py` | RAG管道（YouTube + 新闻） |
| `partition_documents` | `info_partition.py` | 信息不对称创建 |
| Streamlit应用 | `app_consult.py` | 用户界面与会话管理 |

### 7.3 多模型分发

智能体以轮询方式分配到可用的LLM后端：

```python
for i, expert in enumerate(panel):
    backend = backends[i % len(backends)] if backends else None
    agents.append(ConsultAgent(**expert, backend=backend))
```

这提供了：
- **推理多样性**：不同LLM具有不同的推理模式
- **成本优化**：混合高端和经济型模型
- **速率限制管理**：跨服务商分散API调用
- **容错性**：如果一个后端失败，其他后端继续工作

### 7.4 并行执行

所有阶段使用 `ThreadPoolExecutor`，`max_workers=len(agents)` 实现全并行：

```python
with ThreadPoolExecutor(max_workers=len(self.agents)) as pool:
    futures = {pool.submit(_run, a): a for a in self.agents}
    for future in as_completed(futures):
        agent = futures[future]
        results[agent.name] = future.result()
```

完整咨询的典型耗时：2-4分钟（对比串行执行的10-20分钟）。

## 8. 应用案例

### 8.1 加密货币投资分析

**场景**：用户认为"比特币今年能到15万美元，因为ETF资金持续流入和减半效应"。

**专家回应（示例）**：

- **链上数据分析师**："MVRV Z-Score 当前为2.8，表明仍有上行空间，但长期持有者（LTH）供应分布尚未达到典型的周期顶部模式。15万美元的目标需要持续的供应紧缩。"

- **宏观经济学家**："ETF流入是需求端的正面因素，但15万美元意味着全球流动性的大幅扩张，这并非确定。如果通胀反弹，降息预期——比特币的关键驱动因素——可能崩塌。"

- **风险管理顾问**："历史最大回撤为77-93%，意味着你应该问自己：'在通往15万的路上，我能承受50%的跌幅吗？'仓位管理比价格目标更重要。"

- **监管政策专家**："ETF获批是里程碑事件，但SEC对交易所的诉讼仍在进行中，制造了监管不确定性。一个突发的执法行动可能引发20%以上的回调。"

**交付的价值**：用户的论点并非错误，但他们了解到了此前未考虑的供应端约束、宏观依赖性、仓位管理风险和监管尾部风险。

### 8.2 体育预测分析

**场景**：用户认为"湖人肯定能进季后赛，因为他们有LeBron"。

**专家回应（示例）**：

- **数据统计专家**："当前净效率值+1.2意味着约62%的季后赛概率。不是'肯定'——接近4成的模拟显示他们会错过。"

- **伤病体能专家**："AD 27%的生涯缺阵率是关键变量。AD缺阵时，球队净效率值下降10分——从季后赛球队变成乐透球队。"

- **博彩市场分析师**："市场对湖人进季后赛的赔率为-150（隐含概率60%）。如果你认为'肯定'（>90%），市场与你的判断相差30个百分点。"

- **资深体育评论员**："有球星不等于能进季后赛——参见2019年湖人（有LeBron，战绩33胜49负）。体系适配和健康同样重要。"

**交付的价值**：用户的信心从"肯定"（约95%）被校准到更现实的60-65%，并获得了具体的监控变量。

## 9. 评估框架

### 9.1 评估指标

我们提出从三个维度评估该框架：

**1. 观点校准度**
- 测量用户在咨询前后的置信度变化
- 对于可验证的预测：使用Brier Score比较用户咨询后置信度与实际结果
- 目标：咨询后Brier Score < 咨询前Brier Score

**2. 盲点发现率**
- 统计专家识别出的、用户未提及的独特风险因素/考虑事项数量
- 测量用户在第2阶段对话中对新因素的认知程度
- 目标：每次咨询 ≥ 3个新考虑因素

**3. 专家多样性**
- 测量专家间一致程度（越低 = 视角越多元）
- 追踪专家间置信度的离散程度
- 测量立场覆盖度：所有4种立场类型在输出中均有体现
- 目标：没有两位专家给出完全相同的评估

### 9.2 对比基线

| 基线 | 描述 |
|------|------|
| 单一LLM | 同样的问题交给一个LLM实例 |
| 单一LLM + "魔鬼代言人"提示词 | 要求一个LLM挑战用户观点 |
| Level 1 多智能体 | 5个仅有人设描述的智能体（无知识档案） |
| **本系统（Level 2）** | 5个具有知识档案的智能体 + 4阶段流程 |

### 9.3 实验设计（拟定）

1. **数据集**：50个可验证的预测问题（25个加密货币，25个体育），具有已知结果
2. **参与者**：具有预设观点（准确度各异）的模拟用户
3. **指标**：前/后Brier Score、盲点数量、专家多样性指数
4. **假设**：Level 2克隆 + 多阶段流程比所有基线产出更好的校准度

## 10. 局限性与未来工作

### 10.1 当前局限

1. **知识档案的时效性**：知识档案中的少样本示例是静态的，不会随市场条件或新事件更新。

2. **缺乏真值反馈闭环**：系统不追踪其建议是否导致了更好的结果。反馈机制将使Level 3克隆（通过微调）成为可能。

3. **语言**：当前实现以中文为主。国际化将扩大用户群。

4. **专家选择**：面板按领域预定义。理想状态下，系统应根据具体话题动态选择专家。

5. **成本**：5位专家的完整4阶段咨询需要约20次LLM调用。以当前API价格计算，每次咨询成本约$0.02-0.50（取决于模型选择）。

### 10.2 未来方向

**Level 3 克隆**：基于精选专家语料（如Arthur Hayes全部博客文章、Willy Woo全部推文）微调LoRA适配器，实现更深层的行为保真度。

**动态专家选择**：使用分类器分析用户话题，从更大的专家池中自动组合最优专家面板。

**专家持久记忆**：每位专家跨咨询维护记忆——记住过去的预测、追踪其准确率，并相应调整置信度。这与原预测市场系统中的 `ReputationTracker` 相连接。

**认知偏差检测**：自动检测确认偏差（用户只与同意自己的专家互动）、锚定效应（尽管有强有力的反面证据，用户观点不变）和权威偏差（用户只信任置信度最高的专家）。

**多轮咨询**：允许用户在真实世界出现新进展后带着更新的观点回来，创建纵向咨询记录。

**聚合机制**：整合原预测市场系统中的LMSR、BTS和声誉加权聚合，在定性报告之外产出数学上有依据的"共识评分"。

---

## 11. 结论

本文提出了一个多智能体AI咨询框架，将单一LLM建议范式转化为结构化的专家面板体验。通过Level 2 AI克隆（知识档案注入与少样本示例）、4阶段咨询流程（评估→交互→讨论→综合）和设计化多样性（多元的专家立场和信息不对称），系统产出的建议比任何单一AI交互更丰富、更平衡、更可操作。

该框架是领域无关的——虽然我们展示了加密货币和体育两个领域的实现，但架构支持任何专家咨询能增加价值的领域：医疗会诊、法律策略、职业决策、政策分析等。

通过让高质量的多专家咨询对所有人可及，我们的目标是让此前只有拥有昂贵人类专家网络的人才能获得的结构化分析思维变得大众化。

## 附录A：知识档案示例（节选）

**专家**：风险管理顾问
**原型**：Nassim Taleb 的尾部风险思维 + Kelly准则

**核心框架**：
1. 波动率分析（历史波动率、隐含波动率、ATR）
2. 回撤分析（最大回撤、恢复时间）
3. 仓位管理（Kelly准则、VaR/CVaR）
4. 相关性风险（动态相关性、尾部相关性）
5. 流动性风险（买卖价差、市场深度、滑点）
6. 极端场景（黑天鹅事件、交易对手风险）

**少样本示例**（节选）：

> **用户**："我准备把50%的资产配置到比特币"
>
> **专家回应**："让我用风险数据帮你理解这个决定的含义。比特币年化波动率约60-80%（对比标普500约15-18%）。50%配置导致组合波动率约30-40%。历史最大回撤：-93%（2011）、-86%（2014）、-84%（2018）、-77%（2022）。50%配置在-77%回撤下意味着总资产损失约38.5%。Kelly准则建议最优仓位约30%（半Kelly以应对参数不确定性）。建议：最大20-30%，分3-6个月建仓，设定组合层面最大回撤容忍度。"

**典型表达**：
- "让我用数字告诉你这意味着什么"
- "如果你错了，最坏情况是什么？"
- "仓位大小比方向判断重要10倍"
